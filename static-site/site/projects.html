Virtualization · Storage · Business continuity

VMware to Hyper-V without risking shared SAN data

Situation

Production workloads had to move from VMware ESXi to Microsoft Hyper-V. The environment included business-critical virtual machines and SAN-presented storage, so the migration had to protect existing data and preserve a recoverable source state.

Risk

A successful disk conversion would still be a failed migration if it altered a source-owned volume, broke identity or DNS dependencies, prevented an application from starting, or removed the ability to fall back.

Approach

  • Inventoried hosts, virtual machines, datastores, LUNs, networks, and backup state.
  • Mapped ERP, Active Directory, DNS, file-service, and recovery dependencies.
  • Separated source preservation from target preparation.
  • Validated boot, storage, networking, authentication, application behavior, and backup visibility on the target.

Outcome

Business-critical workloads were moved and validated while the source SAN data and recovery path remained controlled.

Windows · Identity · Modernization

Making a legacy Windows environment understandable

Situation

An aging Windows environment had accumulated domain controllers, file shares, print services, backup systems, application dependencies, and incomplete documentation.

Risk

Hidden DNS settings, service accounts, authentication paths, application integrations, and untested recovery assumptions made broad changes unsafe.

Approach

  • Established observed state for identity, DNS, time, policy, files, applications, and backups.
  • Separated immediate stabilization from longer-term modernization.
  • Identified unsupported components and operational single points of failure.
  • Sequenced remediation around dependencies, compatibility, rollback, and validation.

Outcome

The environment became supportable enough for immediate recovery work and a staged modernization plan.

Public project · Containers · CI/CD · Security

Hardening Socket23’s delivery path

Situation

The site’s production path used a mutable image, host bind mounts, root container execution, and a single-replica stop-first update policy. Code changes were not represented by a reliable immutable artifact.

Risk

An image pull or restart could remove the only origin, while configuration drift and bind mounts made repository changes difficult to verify in production.

Approach

  • Built a digest-pinned, non-root, multi-architecture image.
  • Added protected pull-request validation, secret scanning, image scanning, and runtime smoke tests.
  • Changed deployment to two same-node replicas with start-first updates, health checks, and rollback.
  • Kept deployment outbound-only so GitHub does not receive production SSH access.

Outcome

The repository now produces a validated immutable image and the node can adopt approved digests through a bounded pull workflow.

Applied AI · GPU systems · APIs · Operations

Private GPU-backed AI infrastructure

Situation

Local AI work required high-memory GPU hardware, several inference frameworks, model-serving APIs, user interfaces, agent integrations, and security-focused workflows.

Engineering questions

Model compatibility was only one concern. GPU memory, context length, concurrency, storage, authentication, API behavior, observability, and failure recovery also shaped the environment.

Approach

  • Evaluated vLLM, llama.cpp, LM Studio, and Ollama against hardware and API requirements.
  • Integrated model endpoints with Open WebUI and agent workflows.
  • Used containers where they improved repeatability and isolation.
  • Treated access scope, logs, data boundaries, and recovery as infrastructure concerns.

Outcome

A working local environment supports private experimentation, model-serving integrations, and security-focused agent research.
