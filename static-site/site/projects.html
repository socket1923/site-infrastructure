Virtualization · Storage · Recovery

Moving production from VMware to Hyper-V

What needed to happen

Production workloads had to move from VMware ESXi to Microsoft Hyper-V. The virtual machines depended on shared SAN storage, identity, DNS, applications, and backup systems. Converting a disk was only one part of the job.

What could go wrong

A change to the wrong volume, a missing application dependency, or a weak source-state plan could turn a platform migration into a data or recovery problem.

What I did

  • Inventoried hosts, virtual machines, datastores, LUNs, networks, and backup state.
  • Mapped the identity, DNS, file, application, storage, and recovery dependencies.
  • Separated source preservation from target preparation.
  • Validated boot, storage, networking, authentication, application behavior, and backup visibility after the move.

Why I’m proud of it

The workloads moved successfully while the source data and recovery options stayed controlled. The result was not just a running VM; it was a working system with a way back.

Windows · Identity · Modernization

Making an aging Windows environment understandable

What I found

The environment had accumulated domain controllers, DNS settings, file shares, print services, backup systems, applications, service accounts, and incomplete documentation.

Why that mattered

Broad changes were unsafe until the authentication paths, dependencies, unsupported components, and recovery assumptions were visible.

What I did

  • Established observed state for identity, DNS, time, policy, files, applications, and backups.
  • Separated immediate stabilization from longer modernization work.
  • Identified single points of failure and components that could not remain.
  • Sequenced changes around compatibility, rollback, and validation.

Why I’m proud of it

I turned an inherited collection of systems into something that could be reasoned about, supported, and modernized without gambling on assumptions.

Public project · Containers · CI/CD · Security

Rebuilding how this site reaches production

What I started with

The site used a mutable image, host bind mounts, root container execution, and a single replica with stop-first updates. The repository and the running site could drift apart.

What I wanted

A change should be reviewed, tested, built once, identified by digest, adopted by the node without exposing production credentials to GitHub, and rolled back if verification failed.

What I built

  • Protected pull requests with static, configuration, secret, image, architecture, and runtime checks.
  • A digest-pinned, non-root, multi-architecture NGINX image.
  • Two same-node replicas with start-first updates and health checks.
  • An outbound-only pull process that verifies HTTPS and runtime security before accepting the release.

Why I’m proud of it

The public site now follows the same careful-change principles I expect from critical infrastructure. The repository is public, so the implementation can be inspected rather than taken on faith.

Private AI · GPUs · APIs · Operations

Building private AI infrastructure I can operate

What I wanted to learn

I wanted local model access without treating AI as a sealed application. That meant understanding the GPU limits, runtimes, model formats, APIs, interfaces, storage, observability, and agent integrations myself.

What made it interesting

Model quality was only one variable. Context size, concurrency, VRAM, quantization, API behavior, authentication, logs, upgrades, and recovery all shaped the system.

What I built

  • GPU-backed inference across vLLM, llama.cpp, LM Studio, and Ollama.
  • OpenAI-compatible endpoints connected to Open WebUI and agent tools.
  • Containerized components where repeatability and isolation helped.
  • Local retrieval, operational memory, and security-oriented experiments.

Why I’m proud of it

I can test new models and agent ideas privately, understand the infrastructure beneath them, and apply the lessons to safer operational designs.
